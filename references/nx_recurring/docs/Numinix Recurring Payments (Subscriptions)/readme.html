Numinix Recurring Payments (Subscriptions) for Zen Cart

by Numinix


Overview

Numinix Recurring Payments (Subscriptions) extends Zen Cart with automated subscription billing, saved-card renewals, and tools for operators to manage PayPal agreements at scale. The plugin bundles cron utilities, admin dashboards, storefront helpers, and cache infrastructure to keep subscriber data current without slowing down shoppers.

Use this guide to install the plugin, connect the PayPal APIs, configure saved-card flows, automate cache warming, and maintain your recurring portfolio.

Key components

  • cron/ – Billing, reminder, cache refresh, and worker scripts that execute from the command line to process subscriptions on schedule.
  • includes/classes/ – Core PHP orchestrators such as paypalSavedCardRecurring.php that communicate with PayPal, generate orders, and adjust customer group pricing.
  • includes/modules/pages/my_subscriptions/ – Storefront experience and helper functions that hydrate subscription cards, enqueue refreshes, and surface metrics to shoppers.
  • management/ – Admin tools and installers that register menu entries, SQL schema, and configuration settings for saved-card recurring flows.
  • docs/ – Operational notes on cache retention, queue processing, gateway selection, and quality assurance that are consolidated throughout this document.

Optimization workstreams delivered

  • Cache hardening and prefetching – Environment-aware TTLs, invalidation hooks, and queue-based refresh keep paypal_recurring_profile_cache warm without blocking page loads.
  • Gateway selection improvements – New heuristics and persisted hints let PayPalProfileManager choose the correct API client with high confidence while logging fallbacks for tuning.
  • Rendering and UX refactors – Bulk-prefetch helpers, parallel refresh tooling, and UX polishing reduce wait times and improve visibility for both admin and customer subscription pages.
  • Background infrastructure – Queue workers, monitoring hooks, and automation guidance streamline refresh throughput while surfacing actionable metrics to operators.
Installation

Prerequisites

  1. Back up your Zen Cart database and catalog files before deploying the plugin.
  2. Verify your server meets the PHP requirements for PayPal Website Payments Pro (WPP) and the REST SDK (PHP 7.3+, cURL, JSON).
  3. Ensure the PayPal REST PHP library is installed and autoloadable so PayPalRestfulApi classes can be resolved by includes/modules/payment/paypal/pprAutoload.php.
  4. Collect PayPal API credentials for both the classic (NVP/SOAP) and REST clients, along with webhook endpoints for subscription lifecycle events.

Uploading files

  1. Unzip the distribution package locally.
  2. Rename the YOUR_ADMIN/ directory in the package to match your protected admin folder.
  3. Upload the includes/, cron/, management/, and renamed admin directories into your Zen Cart installation, merging when prompted.

Database and configuration bootstrap

  1. Log into Zen Cart admin and navigate to Tools → Install SQL Patches.
  2. Run the installer scripts shipped with the plugin (for example, management/includes/installers/saved_credit_cards_recurring/*.php) if they have not executed automatically.
  3. Confirm the new configuration group Configuration → Saved Credit Cards Recurring is present alongside Customers → PayPal Subscriptions in the menu.

PayPal REST setup

  1. Open Configuration → PayPal and enter the REST Client ID, Client Secret, and environment constants used by MODULE_PAYMENT_PAYPALR_* definitions.
  2. Register PayPal webhooks for BILLING.SUBSCRIPTION.*, PAYMENT.SALE.*, and VAULT.PAYMENT-TOKEN.*, pointing to the same endpoint handled by the PayPal REST payment module so saved-card edits and reminders stay in sync.
  3. Verify cron users share access to the same credentials and SDK files so background jobs can hydrate REST profiles without falling back to legacy flows.

Once files and configuration are in place, the admin menu exposes Customers → PayPal Subscriptions for operators and the storefront renders a refreshed My Subscriptions page for customers.

Configuration

Configuration spans classic PayPal subscriptions, REST saved-card renewals, and the cache infrastructure that supports both. Review each group after installation.

  1. Saved Credit Cards Recurring

    • Enable or disable recurring billing globally and choose whether new subscriptions default to PayPal REST (paypalr) or classic WPP depending on your merchant account capabilities.
    • Set retry rules, reminder schedules, and debug logging destinations to control how aggressively the system follows up on missed payments.

  2. PayPal REST credentials

    • Populate MODULE_PAYMENT_PAYPALR_CLIENT_ID, MODULE_PAYMENT_PAYPALR_CLIENT_SECRET, and MODULE_PAYMENT_PAYPALR_MODE. Missing credentials force the module to skip REST-only features such as inline card updates.
    • Map webhook events to the profile manager so subscription edits and payment notifications remain synchronized across the storefront, admin, and cron contexts.

  3. Profile cache retention

    • Review the automatic TTL applied based on environment keywords (sandbox, staging, production). Override the TTL with ZEN_PAYPAL_SUBSCRIPTION_PROFILE_CACHE_TTL if local requirements differ.
    • Confirm cleanup jobs align with your data retention policies; the plugin purges rows older than the environment-specific horizon and resets the in-request cache to prevent stale reads.

  4. Gateway hinting and fallbacks

    • Leave preferred_gateway persistence enabled so future profile lookups reuse the last known-good API client without re-running heuristics.
    • Monitor the admin logs for fallback events; repeated fallbacks indicate mismatched credentials or stale cache records that may need manual refresh.

  5. Debug logging

    • Enable My Subscriptions Debug Logging to emit timing, query counts, and API responses to DIR_FS_LOGS . 'my_subscriptions_debug.log' when troubleshooting storefront latency.
    • Define MY_SUBSCRIPTIONS_DEBUG and MY_SUBSCRIPTIONS_DEBUG_LOG in includes/configure.php to override the admin toggle for ad-hoc investigations.
Usage

Once credentials are verified, operators and customers can manage subscriptions without leaving Zen Cart.

  1. Admin: Customers → PayPal Subscriptions

    Administrators can edit active profiles inline. Legacy NVP/SOAP agreements continue to expose the full recurring profile form, while REST profiles support amount, payment source, and next-billing-date adjustments via PayPalProfileManager. Validation errors from PayPal surface directly in the UI so staff can retry quickly.

  2. Admin: Monitor refresh health

    Run Customers → PayPal Subscriptions alongside queue metrics to review pending refreshes, queue depth, and last-run timestamps. The grid highlights stale entries so operators can trigger manual refreshes or investigate PayPal API issues.

  3. Storefront: My Subscriptions

    The storefront page consumes cached classifications to render subscription cards quickly. Helper functions batch fetch cache rows, hydrate plan details, and respect manual refresh flags without issuing redundant API calls.

  4. Direct refresh controls

    Operators can enqueue specific subscriptions for refresh via the admin or storefront triggers. Queue helpers manage locking, retries, and context metadata so background workers can hydrate both paypal_recurring_profile_cache and paypal_recurring entries efficiently.

Retired workflows

Legacy customer-driven refresh buttons have been removed. Subscription status now updates through PayPal IPN notifications, admin edits, or background queue processing. Ensure any historic monitoring or automation relying on the retired endpoints is cleaned up to avoid confusion.

Automation

Schedule cron jobs to process renewals, reminders, and cache refreshes without manual intervention. All scripts assume your Zen Cart root is the working directory.

Core jobs

  • php cron/paypal_wpp_recurring_billing.php – Captures due subscriptions using the configured gateway and creates new Zen Cart orders.
  • php cron/paypal_wpp_recurring_reminders.php – Emails reminder notices and logs retry attempts for delinquent accounts.
  • php cron/paypal_profile_cache_refresh.php – Scans for stale cache rows, enqueues up to 100 refresh jobs per run, and emits queue-depth metrics to stdout.
  • php cron/paypal_profile_refresh_worker.php [batchSize] – Claims queued refresh jobs, hydrates profile cache data, and records success/failure statistics for monitoring.

Sample crontab entries

/etc/cron.d/zen-cart-recurring
*/15 * * * * www-data /usr/bin/php /var/www/html/cron/paypal_wpp_recurring_billing.php
0 * * * * www-data /usr/bin/php /var/www/html/cron/paypal_wpp_recurring_reminders.php
*/10 * * * * www-data /usr/bin/php /var/www/html/cron/paypal_profile_cache_refresh.php
*/5 * * * * www-data /usr/bin/php /var/www/html/cron/paypal_profile_refresh_worker.php 200

Adjust cadences to match subscription volume and API rate limits. Running multiple workers in parallel significantly decreases refresh wall-clock time; staging benchmarks showed three workers processing 90 profiles in 8 minutes 45 seconds versus 27 minutes sequentially.

Uninstall

Follow these steps to safely remove the plugin while retaining historical subscription data for audit purposes.

  1. Prepare the store

    Disable cron entries related to PayPal recurring billing, reminders, cache refresh, and queue workers. Inform staff that subscription management will be paused during removal.

  2. Disable modules

    In Zen Cart admin, set the Saved Credit Cards Recurring configuration group to disabled and remove menu permissions for Customers → PayPal Subscriptions so operators cannot initiate new edits during teardown.

  3. Run uninstall SQL

    Use Tools → Install SQL Patches to execute the following script. The destructive DROP TABLE statements are commented out to protect data; uncomment them only if you intend to remove subscription history.

    uninstall_paypal_recurring.sql
    -- Remove configuration and admin page registrations
DELETE FROM configuration WHERE configuration_key LIKE 'MODULE_PAYMENT_PAYPALR_%';
DELETE FROM configuration WHERE configuration_key LIKE 'MY_SUBSCRIPTIONS_DEBUG%';
DELETE FROM configuration_group WHERE configuration_group_title = 'Saved Credit Cards Recurring';
DELETE FROM admin_pages WHERE page_key IN ('paypalsubs', 'configSavedCreditCardsRecurring');

-- Optional: drop cache/queue tables (uncomment to delete data)
-- DROP TABLE IF EXISTS paypal_recurring_profile_cache;
-- DROP TABLE IF EXISTS paypal_recurring_refresh_queue;
-- DROP TABLE IF EXISTS paypal_recurring; -- removes all subscription records

  4. Remove files

    Delete plugin-specific directories from includes/modules/payment/, includes/classes/, includes/modules/pages/my_subscriptions/, cron/, and management/. Clear any cached templates referencing the subscription helpers.

  5. Validate cleanup

    Load the storefront and confirm the My Subscriptions page is no longer accessible. Review cron logs to ensure no recurring scripts continue to run, and verify there are no fatal errors referencing removed classes.

Troubleshooting
  • REST updates are skipped? Confirm the PayPal REST SDK is installed and credentials are configured. When credentials are missing the module logs failures and falls back to legacy pathways.
  • Customer page is slow? Enable My Subscriptions debug logging to capture timing details and confirm cache prefetching is active. Look for repeated cache misses or slow API responses in my_subscriptions_debug.log.
  • Queue backlog growing? Review paypal_profile_refresh_worker.php logs for repeated failures and inspect the paypal_recurring_refresh_queue table for stuck locks or high retry counts. Increase worker concurrency or adjust TTL thresholds if pending jobs continually exceed capacity.
  • Gateway mismatch errors? Clear or refresh cache rows where preferred_gateway no longer matches the active API credentials. Re-run the gateway heuristics by forcing a refresh from the admin grid.
  • Legacy refresh tools appear? Ensure obsolete automation referencing the retired customer refresh workflow has been removed so templates and cron jobs do not reintroduce deprecated UI elements.
